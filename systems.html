05 / Systems The layers I return to

Realtime. /
Offline-first. /
AI-augmented.

Across six companies, I keep architecting the same three technical layers: live media, offline data sync, and AI content pipelines. Here is how I approach these systems, and the hard lessons learned from running them in production.

05.01 / Realtime & Media Socket.IO · WebRTC · Mediasoup · Agora

Live media: / where fragile / architectures break.

The complexity of real-time systems doesn't lie in the SDK integrations—it lies in managing the in-between states. What happens when a participant drops due to network jitter? How does Picture-in-Picture (PiP) behave when the OS terminates your activity? Who truly owns the signaling contract if you need to swap the SFU later? These are the edge cases I engineer for.

I have architected four distinct real-time stacks in production. I deployed a Mediasoup SFU at Medjeex for 30-student live classes to maintain total signaling control. I integrated Agora RTC for Ayla Voice Chat's multi-user audio rooms. I built WebRTC with custom signaling for QuickDock's telemedicine consults. I implemented PiP-mode adaptive streaming for JagoBd's 10M+ Android TV/mobile install base. Every stack was chosen deliberately to optimize for cost, control, latency, or platform reach.

Where I've shipped this
  • Medjeex live classes — Mediasoup SFU + flutter_webrtc + Socket.IO, PiP, foreground service.
  • Ayla Voice Chat — Agora RTC multi-user voice rooms + Socket.IO messaging.
  • QuickDock — WebRTC video consults, doctor-availability scheduling.
  • JagoBd Android TV — PiP adaptive streaming on 10M+ install base.
  • Bodhi astrology — Socket.IO live-availability indicators and chat with astrologers.
05.02 / Offline-first Isar · Repository · AES

Treat local / as truth. The / API is a peer.

Most applications treat the backend API as the absolute source of truth and the local database as a fragile cache. That mental model fails the moment the network drops. I build systems that treat the local database as the truth, and the API as merely one of many peers allowed to mutate it. The UI never cares if the data is fresh—managing that state is strictly the Repository layer's job.

At Medjeex, this manifests as a self-test engine that securely pre-downloads 500-question packs, grades them deterministically offline, and syncs the results via a replayable background queue. It manifests as AES-encrypted study material that students can reliably access on a train. It manifests as Isar schemas that migrate seamlessly across years of product evolution.

Where I've shipped this
  • Medjeex self-test engine — ahead-of-time download, local grading, post-session sync.
  • Medjeex notes & PDFs — AES-encrypted offline delivery via Syncfusion PDF viewer.
  • FuturePeak MCQ portal — offline question packs for on-demand testing.
  • Course downloads — encrypted offline lecture access with re-validation on reconnect.
05.03 / AI-augmented UX RAG · prompt engineering · content pipelines

Generate / once. Consume / infinitely.

A naive AI wrapper makes an LLM API call every time a user taps a button. In this model, operating costs scale linearly with active users, latency degrades with model size, and the UI is forced to show loading spinners. Instead, I architect AI content pipelines that generate artifacts once on the server during ingestion, allowing the mobile clients to consume them instantly as cached static assets.

At Medjeex, I engineered a pipeline that processed 2,000+ lecture recordings in a single pass: transcribe → summarize → extract MCQs → extract flashcards → synthesize two-voice podcast TTS. When a student opens a lecture, the assets are already compiled, instantly available, and ready for offline use. The only live LLM inference in the entire app is the RAG-powered ai_test_mentor that explains missed questions—and even that is strictly bounded, cached, and optional.

Where I've shipped this
  • Medjeex content pipeline — 2,000+ lectures → notes / MCQs / podcast TTS.
  • ai_test_mentor module — RAG-style explainer + quiz loop inside test analysis.
  • Doubt module first-pass AI — AI responds first, peer/teacher escalation.
  • Two-voice podcast TTS — just_audio player for AI-generated lecture podcasts.
  • Krishworks Speaklifi — AI voice-recognition feedback loops for pronunciation learning.
05.04 / Manifesto What I won't do

Things I won't / build, even if / asked.

01

Chat wrappers

If the whole feature is a textbox sending raw prompts to an LLM, that's not a feature, that's a bill.

02

Fake metrics

No "improved engagement 32%" without a dashboard I can show. The receipts have to hold up.

03

Over-architecture

Clean Arch with seven layers for a CRUD screen. Repository pattern when there's one source of data. Pick the layers that earn their seat.

04

Animation theatre

Springy bouncy parallax that ages the app in six months. Motion has to serve content, not the designer's reel.
