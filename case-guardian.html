Case 02 Medjeex Guardian

A second / screen. / For the / parent.

Medjeex Guardian is the parent-facing companion to the Medjeex student platform — a separate Flutter application that gives parents full visibility into their child's attendance, test performance, and live study activity, without touching the student's own app experience.

Role Sole Flutter developer
Year 2025
Stack Flutter · Provider · ChangeNotifierProxyProvider · go_router · fl_chart · FCM · flutter_secure_storage · Isar
Domain EdTech · Companion / monitoring app
Live Play Store ↗
01 / The decision

Separate / app, / not a / mode.

Most platforms bolt a parent view onto the student app — a settings screen, a web portal, a restricted login. That approach saves time but creates a compromised experience for both users: the student's app grows heavier, and the parent gets a surface that was never designed for them.

The decision here was different: build a separate Flutter application that connects to the same production backend at api.medjeex.com through a dedicated parent-scoped API surface. The parent gets a first-class product. The student app stays lean. The backend enforces the boundary — a parent cannot query a child's dashboard without an active link that the student themselves approved.

02 / Architecture

Core / features. / Mirrored / deliberately.

The codebase follows the same core/features split as the main Medjeex student app — a stated constraint documented in docs/implementation_plan.md. The core/ layer provides the ApiClient, AppRouter, a multi-layered design token system (app_colors, app_typography, app_spacing, app_shadows, app_breakpoints), and shared services. Six feature verticals sit under features/: auth, child, dashboard, home, notifications, and profile.

Every feature follows the same three-layer pattern: models/ (plain Dart data classes with fromJson factories), repositories/ (raw API call logic), and viewmodels/ (ChangeNotifier state containers). This consistency means any engineer familiar with the student app can navigate the parent app without a re-orientation period — which is precisely the point of mirroring.

03 / The hard problem Multi-child context

One parent. / Multiple / children. / One active / context.

01

ProxyProvider

DashboardViewModel is registered as a ChangeNotifierProxyProvider<ChildViewModel, DashboardViewModel>. When the active child changes, the dashboard re-fetches automatically — zero manual orchestration in the UI.

02

Per-child cache

A Map<String, DashboardSummary> keyed by child ID. Switching from Child A to Child B doesn't discard Child A's data — it stays cached for instant restoration on the next switch.

03

Parallel fan-out

Future.wait fires 6 simultaneous API calls — attendance, watch time, DPP status, test performance, test trend, test history — rather than chaining them serially.

04 / Consent gate

The OTP / goes to / the child, / not the / parent.

When a parent initiates a child-link by entering the student's phone number, the backend sends the OTP to the student's phone. The parent must collect that code from the child and submit it to complete the link.

This is not a UX quirk — it's a deliberate consent gate. A student cannot have their academic data exposed to a parent account without knowing about it and actively participating. The LinkedChild model captures the lifecycle with a status field (pending, active, revoked) and a linkedVia field. The student retains the ability to revoke the link at any time.

The backend enforces this at the data layer too — a parent's JWT is scoped to the parent user, and the server validates that the requesting parent is actually linked to the queried child before returning any dashboard data. The mobile client cannot sidestep this even with a valid token.

06 / Engineering decisions
State · ProxyProvider pattern
Reactive dashboard, zero manual wiring
Rather than having the UI trigger a dashboard reload on child switch, DashboardViewModel is declared as dependent on ChildViewModel via ChangeNotifierProxyProvider. The framework calls updateActiveChild() automatically whenever ChildViewModel notifies — making the active child the single source of truth without any coordinator logic in the view layer.
Navigation · go_router + StatefulShellRoute
Tab state preserved independently
StatefulShellRoute.indexedStack gives each of the three bottom-nav tabs its own NavigatorKey. Switching from Analytics back to Home does not destroy and recreate the Home widget tree — scroll position and loaded data survive. Deep-drill views (test details, attendance, offline reports) are registered on the root navigator key, rendering modally above the bottom bar rather than inside a tab.
Data · Dual attendance model
Online lectures and offline check-ins in one model
The Medjeex platform supports both online-only and hybrid (physical coaching centre) delivery. AttendanceReport carries both online fields (totalScheduled, totalAttended, currentStreak) and offline fields (totalPresent, totalAbsent, checkIn/checkOut timestamps). The backend switches response shape based on the course's deliveryMode, and the model handles both without conditional branching in the UI.
Notifications · FCM + SharedPreferences ring buffer
Persistent notification inbox, no extra database
All incoming FCM messages (foreground, background, and app-launch) are written to a capped 100-item ring buffer in SharedPreferences by a @pragma('vm:entry-point') background handler — the required pattern for Flutter's FCM isolate model. NotificationViewModel reads this same key and exposes an unreadCount getter for the bell badge. FCM token registration is deduplicated by hash — the token only syncs to the backend when it changes, avoiding a race with the auth flow on startup.
