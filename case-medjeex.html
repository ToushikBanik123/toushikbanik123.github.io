03.01 / Problem Where we started

Zero to One. / Scale to / Thousands.

Medjeex required a robust mobile ecosystem for NEET and JEE aspirants. The constraints were severe: students often learn from remote areas on intermittent 4G networks. I was tasked with engineering four distinct applications from scratch—a primary student app, a guardian companion, a realtime attendance scanner, and a teacher console. There was no legacy code, no inherited architecture. The technical bar was non-negotiable: test sessions must survive network drops, and live video must degrade gracefully without killing the main thread.

03.04 / Pillars 06 systems

What's running / inside.

03.04.01

Live video classes

mediasfu_mediasoup_client (a maintained fork picked for WebRTC compatibility) + flutter_webrtc inside Flutter, with Socket.IO signalling. The meetings/ module handles participant lobby admission, host admit/remove controls, in-meeting chat and native Picture-in-Picture. An Android foreground service (meeting_foreground_service.dart) keeps the call alive when the app is backgrounded — otherwise the call drops the moment a student switches apps.

03.04.02

Offline self-test engine

The self_test/ module lets students create a test by subject + chapter + difficulty + count. Questions are Isar-cached ahead of time so the entire test runs locally — MCQ, MSQ, integer-answer, all timed. Grading is deterministic and computed on-device, then synced to the backend post-session. result_comparison_card.dart renders comparative analytics versus prior attempts.

03.04.03

AI mentor + content pipeline

2,000+ lecture recordings are processed once at upload-time: transcript → notes → MCQs → flashcards → two-voice podcast TTS. The ai_test_mentor/ module is a separate RAG-style flow inside test analysis — when a student misses a question, it explains the underlying concept, suggests the relevant lecture clip, and then quizzes them on it. Everything ships as static artefacts; nothing is generated per tap.

03.04.04

Doubt resolution

Tiered escalation: AI responds first inside the doubt/ module, peers can upvote or escalate, teachers fall back in if the question gets stuck. Nested threaded comments, filtering, read-caching. My-doubts / saved-doubts / delete-request flows for the student-side of the surface.

03.04.05

Course delivery

Course → Chapter → Lecture hierarchy. Live lectures via the meetings stack with Socket.IO chat. Recorded lectures via youtube_player_flutter + just_audio for the AI podcast track. Syncfusion PDF viewer for notes with AES-encrypted offline delivery. Deep-link sharing via app_links so a teacher can drop a link in a WhatsApp group and a student lands on the right lecture.

03.04.06

Auth, payments & release

JWT + OTP authentication built with pinput and sms_autofill, BCrypt credential hashing, flutter_secure_storage for token persistence. AES-encrypted offline PDFs. razorpay_flutter wired through cart, coupon and wishlist flows. shorebird_code_push for OTA — same-day Dart hotfixes to 1,000+ active users without Play Store review.

03.05 / Decisions 06 trade-offs

Why I picked / what I picked.

Mediasoup SFU / vs Agora RTC

Why I built the video stack on Mediasoup, not bought it.

Agora was the obvious managed pick — I'd already shipped voice rooms on it at Ayla. But at ~30-student class sizes Medjeex needed signalling control we'd never get from a managed SDK: lobby admission, host admit/remove, deterministic PiP behaviour on Android. I went with mediasfu_mediasoup_client (a maintained fork picked for WebRTC compatibility) over the upstream — owned the Socket.IO signalling surface end-to-end, predictable cost at scale, Flutter binding mature enough to ship.

Isar / vs SQLite

Why Isar runs the offline test engine.

The test engine downloads 500-question packs ahead of time and grades fully locally. Isar's collection-level encryption protects question banks at rest, schema-migration ergonomics matched how often we iterated, and bulk reads on those packs were faster than SQLite for the access pattern we actually had — single-table sequential scans of cached questions.

Shorebird OTA / vs Play Store rollouts

Why same-day hotfix latency was non-negotiable.

A paid 1,000-user EdTech app cannot wait two days for a Play Store review when a live class breaks. Shorebird Flutter code-push gave us hotfix-to-device in hours instead of review cycles. The trade-off — runtime-only Dart fixes, no native — was acceptable because most of our regressions lived in UI/state logic, exactly where Shorebird shines.

Offline-first / over Online-with-cache

Why the Repository was the source of truth, not the API.

NEET/JEE students take tests on patchy networks. Treating the API as truth meant flaky exams; treating Isar as truth meant deterministic grading even when the device went offline mid-test, with a sync queue that replayed to the backend on reconnect. The UI never knows whether data is fresh — that's the Repository's job.

Server-side pipeline / vs On-device LLM

Why notes / MCQs / podcast are generated once, not per tap.

Generating per-tap meant unpredictable cost per active student and 5-second waits in the UI. Running the pipeline once per lecture upload — transcribe, summarise, MCQ-extract, two-voice TTS — let students treat AI artefacts like any other static asset: cached, instant, downloadable. The mobile app stays a thin consumer of a structured content store.

22 feature modules / over One big lib/

Why feature modules paid off as the solo dev.

Working alone meant context-switching was the real cost, not coordination. Splitting lib/features/ into 22 modules (auth, meetings, self_test, course, doubt, ai_test_mentor, live_quiz, pomodoro, routine, store, quest, …) let me touch one surface without paging the rest into my head, kept compile times sane, and made shared core/ primitives reusable across the student, parent, attendance and teacher apps.
